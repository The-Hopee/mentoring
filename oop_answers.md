# Подготовка к собеседованию на Middle C++. Часть 2

## 1) Общие знания по ООП

### 1.1)  Что такое ООП?
Это парадигма программирования, в которой программа представляется как совокупность взаимодействующих объектов, объединяющих данные и поведение. ООП фокусируется на объектах, а не на логике или функциях.

### 1.2)  Что такое класс?
Класс — это пользовательский тип данных, чертёж или шаблон, описывающий структуру и поведение будущих объектов. Он инкапсулирует данные (поля) и функции (методы) для работы с ними.

### 1.3)  Что такое объект?
`Объект` — это конкретный экземпляр класса, обладающий собственным состоянием (значениями полей) и поведением, определённым классом. Память под объект выделяется во время выполнения программы.

### 1.4) Расскажи про основные принципы ООП: абстракция, наследование, полиморфизм, инкапсуляция.
`Абстракция` — выделение существенных характеристик и игнорирование несущественных деталей реализации. `Инкапсуляция` — сокрытие внутреннего состояния и деталей реализации от внешнего мира с предоставлением контролируемого доступа через публичный интерфейс. `Наследование` позволяет создавать новые классы на основе существующих (иерархия), повторно используя и расширяя их функциональность. `Полиморфизм` — способность функции или метода обрабатывать данные разных типов, единый интерфейс для различных реализаций (виртуальные функции и перегрузка).

### 1.5)  Что такое АБ? Что такое интерфейс?
`Абстрактный базовый класс (АБК)` — это класс, который не может быть инстанциирован и содержит хотя бы одну чистую виртуальную функцию (`= 0`). `Интерфейс` — это понятие, описывающее набор публичных методов без реализации; в C++ роль интерфейса часто играет АБК без данных и с одними чистыми виртуальными функциями.

### 1.6)  Расскажи про принципы SOLID.
`S` — Класс должен иметь одну ответственность. `O` — Классы открыты для расширения, но закрыты для модификации. `L` — Подтипы должны заменять базовые типы без изменения корректности программы. `I` — Много специализированных интерфейсов лучше одного общего. `D` — Модули верхнего уровня не должны зависеть от деталей; и те, и другие должны зависеть от абстракций. S — Single Responsibility Principle (принцип единственной ответственности). O — Open/Closed Principle (принцип открытости/закрытости). L — Liskov Substitution Principle (принцип подстановки Барбары Лисков). I — Interface Segregation Principle (принцип разделения интерфейса). D — Dependency Inversion Principle (принцип инверсии зависимостей).

## 2) Методы класса

### 2.1)  Что такое конструктор?
Это специальный нестатический метод класса, автоматически вызываемый при создании объекта для его инициализации. Конструктор не имеет типа возврата и имя конструктора совпадает с именем класса.

### 2.2)  Зачем используется ключевое слово `explicit`?
`explicit` запрещает неявный вызов конструктора с одним параметром (конвертирующего конструктора) или оператора преобразования. Это предотвращает случайные, потенциально ошибочные неявные преобразования типов.

### 2.3) Существует ли виртуальный конструктор?
Нет, виртуальных конструкторов в C++ не существует. На момент вызова конструктора таблица виртуальных функций ещё не сформирована (объект ещё не создан), поэтому полиморфизм невозможен.

### 2.4)  Какие виды конструкторов вы знаете?
`Конструктор по умолчанию` (без параметров), `параметризованный`, `копирующий` (принимает `const T&`), `перемещающий` (принимает `T&&`), `конвертирующий` (с одним параметром, кроме `T` и `T&&`). Также `делегирующий` конструктор вызывает другой конструктор того же класса.

### 2.5)  Что такое виртуальная функция?
Это метод класса, объявленный с ключевым словом `virtual`, вызов которого разрешается динамически (во время выполнения) на основе реального типа объекта. Это позволяет через указатель или ссылку на базовый класс вызвать реализацию из производного класса.

### 2.6) Что такое vtable? Расскажи, как она работает.
Это скрытая статическая таблица виртуальных функций, создаваемая компилятором для каждого полиморфного класса. Каждый объект такого класса хранит невидимый указатель на `vtable` своего класса `(vptr)`; при вызове виртуальной функции компилятор переходит по `vptr` к таблице и вызывает нужную функцию по индексу.

### 2.7)  Зачем используется ключевое слово `virtual`?
Для объявления виртуальных функций в базовом классе, что включает механизм динамического полиморфизма. В C++ `virtual` также используется при виртуальном наследовании для решения проблемы ромба, гарантируя единственный экземпляр общего базового класса.

### 2.8)  Зачем используется ключевое слово `final`?
На виртуальной функции запрещает её переопределение во всех дальнейших производных классах. На классе запрещает наследование от этого класса.

### 2.9)  Зачем используется ключевое слово `override`?
Явно указывает, что виртуальная функция в производном классе должна переопределять виртуальную функцию базового класса. Компилятор выдаст ошибку, если подходящей функции в базовом классе нет (например, из-за опечатки в сигнатуре), что спасает от труднонаходимых ошибок.

### 2.10) Что такое чистая виртуальная функция?
Это виртуальная функция, объявленная с инициализатором `= 0` и не имеющая определения (хотя может его иметь). Класс, содержащий хотя бы одну такую функцию, становится абстрактным, и создать его экземпляр нельзя.

### 2.11) Как получить ошибку `pure virtual call`? Напиши код.
Вызвать чистую виртуальную функцию из конструктора или деструктора базового класса (напрямую или косвенно), когда объект производного класса ещё не создан или уже разрушен. В этот момент vptr указывает на vtable базового класса, где функция помечена как чистая.
```cpp
    struct Base {
        Base() { call(); } // Вызов из конструктора
        virtual void pure() = 0;
        void call() { pure(); }
    };
    struct Derived:Base { void pure() override {} };

    int main() { Derived d; } // pure virtual call
```

### 2.12) Что такое деструктор?
Это специальный метод класса, автоматически вызываемый при уничтожении объекта для освобождения захваченных ресурсов (память, файлы, сокеты). `Имя деструктора: ~ClassName()`, он не имеет параметров и типа возврата.

### 2.13) Зачем нужен виртуальный деструктор?
Чтобы гарантировать корректный вызов цепочки деструкторов производных классов при удалении объекта через указатель на базовый класс. Без виртуального деструктора delete base_ptr вызовет только деструктор базового класса, что приведёт к утечке ресурсов.

### 2.14) В чем особенность константных методов класса?
Метод, объявленный с квалификатором `const` (после списка параметров), гарантирует, что он не изменяет состояние объекта (его нестатические поля). Такой метод можно вызывать у константных экземпляров класса или через константные ссылки/указатели. Исключение составляют поля, объявленные с ключевым словом mutable — их разрешено изменять даже в константных методах (например, для кеширования или мьютекса).

### 2.15) Расскажи Правило 5 (Правило 3, Правило 0). Напиши код.
`Правило 3:` Если классу нужен явный деструктор, копирующий конструктор или оператор присваивания копированием, то ему, скорее всего, нужны все три.
`Правило 5:` С введением семантики перемещения, помимо трёх, нужно также определить перемещающий конструктор и оператор присваивания перемещением. 
`Правило 0:` Не управляйте ресурсами вручную; используйте умные указатели и контейнеры, позволяя автоматически генерируемым методам работать корректно.
`Код:`
```cpp
    class Rule5 {
        int* data;
    public:
        Rule5(int v) : data(new int(v)) {}
        ~Rule5() { delete data; }                 // Деструктор
        Rule5(const Rule5& o) : data(new int(*o.data)) {} // Копирование
        Rule5& operator=(const Rule5& o) { if(this!=&o) *data=*o.data; return *this; }
        Rule5(Rule5&& o) noexcept : data(o.data) { o.data=nullptr; } // Перемещение
        Rule5& operator=(Rule5&& o) noexcept { if(this!=&o) { delete data; data=o.data; o.data=nullptr; } return *this; }
    };
```

### 2.16) Расскажи об использовании static для полей и методов класса.
`Статическое поле (static int count;)` — это поле, существующее в единственном экземпляре для всего класса, а не для каждого объекта. `Статический метод (static void foo())` не имеет скрытого указателя this и может обращаться только к статическим полям и другим статическим методам класса.

### 2.17) Расскажи об использовании inline для полей и методов класса.
С C++17 `inline` для статических полей позволяет инициализировать их прямо внутри класса, избавляя от необходимости выносить определение в .cpp-файл. Для методов определение функции внутри тела класса неявно делает её `inline`, что разрешает множественное определение в разных единицах трансляции.

### 2.18) Зачем используется ключевое слово mutable (для полей класса, для лямбды)?
Для полей класса `mutable` позволяет изменять их даже в константных методах (флаг кеширования или мьютекс). В лямбдах `mutable` снимает константность с оператора вызова (`operator()`), разрешая модифицировать захваченные по значению переменные внутри тела лямбды.

## 3) Семантика перемещения

### 3.1) Что такое семантика перемещения?
Это механизм передачи ресурсов от одного объекта другому вместо глубокого копирования. Перемещение «вырывает внутренности» у объекта-источника (`rvalue`), оставляя его в валидном, но неопределённом состоянии, что значительно эффективнее для временных объектов.

### 3.2) Что такое deep copy и shallow copy?
`Shallow copy` (поверхностное) копирует значения полей как есть, включая указатели, что приводит к совместному владению одними и теми же данными в динамической памяти. `Deep copy` создаёт полную независимую копию всех данных, рекурсивно копируя объекты, на которые указывают указатели.

### 3.3) Что такое rvalue и lvalue?
`lvalue` (`left value`) — это выражение, имеющее идентифицируемое место в памяти (имя переменной), можно взять адрес. `rvalue` (`right value`) — это временное выражение, не имеющее постоянного адреса в памяти (литералы, результаты вычислений, временные объекты).

### 3.4) Что делает std::move? Как устроен std::move?
`std::move` выполняет безусловное преобразование (`cast`) переданного аргумента в `rvalue`-ссылку (T&&). По сути это `static_cast<std::remove_reference_t<T>&&>(arg)`, который позволяет выбрать перемещающую перегрузку функции, сам `std::move` ничего не перемещает.

## 4) Исключения

### 4.1) Расскажи, как работает механизм исключений.
При возникновении ошибки вызывается `throw`, создающий объект исключения. Исполнение передаётся от места броска до ближайшего подходящего `catch-блока`, при этом автоматически вызываются деструкторы созданных на пути локальных объектов (раскрутка стека).

### 4.2) Расскажите о иерархии расположения catch-блоков.
`Catch-блоки` обрабатываются последовательно, срабатывает первый подходящий по типу. Блоки для обработки более производных типов (исключений-потомков) должны располагаться раньше блоков для базовых типов, иначе базовый `catch` «перехватит» все исключения, сделав наследников недостижимыми.

### 4.3) Можно ли бросать исключение из конструктора? О чем нужно помнить, когда мы бросаем исключения из конструктора?
Да, бросать исключение из конструктора можно — это единственный способ сигнализировать об ошибке создания объекта. Важно помнить, что объект не будет считаться созданным, его деструктор не вызовется, но деструкторы всех полностью созданных членов и базовых классов будут корректно вызваны.

### 4.4) Можно ли бросать исключение из деструктора?
Категорически не рекомендуется, а начиная с `C++11` деструкторы неявно помечены `noexcept`. Если исключение покинет деструктор во время раскрутки стека из-за другого исключения, это приведёт к аварийному завершению программы (`std::terminate`).

### 4.5) Что произойдет, если std::dynamic_cast не сможет скастовать тип?
При касте ссылки будет выброшено исключение `std::bad_cast`, так как ссылка не может быть `«пустой»`. При касте указателя `dynamic_cast` вернёт нулевой указатель (`nullptr`).

### 4.6) Зачем используется ключевое слово noexcept?
Указывает, что функция не бросает исключений, что является гарантией компилятору и частью интерфейса. Это позволяет компилятору генерировать более оптимальный код, а библиотекам (например, `std::vector`) использовать более эффективные стратегии (перемещение вместо копирования при реаллокации). Кроме спецификатора, есть ещё оператор `noexcept(expr)`, который на этапе компиляции проверяет, считается ли выражение невыбрасывающим, и возвращает `true` или `false` и в случае `true` помечает функцию как `noexcept`  

## 5) Идиомы

### 5.1) Что такое идиома RAII? Напиши код. Приведите примеры RAII в STL.
`RAII (Resource Acquisition Is Initialization)` — захват ресурса есть инициализация. Ресурс (память, файл, мьютекс) захватывается в конструкторе и гарантированно освобождается в деструкторе, что исключает утечки даже при возникновении исключений. Примеры STL: std::unique_ptr, std::lock_guard, std::ifstream.
`Пример кода`
class jthread
{
private:
    std::thread t;
public:
    jthread( std::thread t_): t(std::move(t_))
    
    ~jthread
    {
        if(t.joinable()) j.join()
    }
}

### 5.2) Что такое RVO? Напиши код.
`Return Value Optimization` — гарантированная с C++17 оптимизация, при которой временный объект, создаваемый в return, конструируется сразу по адресу возвращаемого значения в вызывающем контексте.
std::string create() { return std::string("hello"); } // RVO: временный объект не копируется.

### 5.3) Что такое NRVO? Напиши код.
`Named Return Value Optimization` — необязательная оптимизация, исключающая копирование именованного локального объекта при его возврате из функции. Компилятор конструирует этот объект прямо в области памяти вызывающей функции.
std::string create() { std::string s("hello"); return s; } // NRVO исключает копирование s.

### 5.4) Что такое SSO?
`Small String Optimization` — реализация `std::string`, при которой короткие строки (обычно до 15 символов) хранятся во внутреннем буфере самого объекта (на стеке), а не выделяются в динамической памяти. Это повышает производительность и снижает фрагментацию памяти для небольших строк.

## 6) Разное

### 6.1) В чем отличие между классом и структурой?
Фактическое различие только одно: в `struct` модификатор доступа по умолчанию для членов — `public`, для базовых классов — `public`; в class — `private`. Семантически принято использовать `struct` для пассивных агрегатов данных (POD), а `class` — для инкапсулированных объектов с инвариантами.

### 6.2) Расскажи, как устроена лямбда.
Компилятор генерирует анонимный класс с оператором `operator()` (метод с телом лямбды) и набором полей для захваченных переменных. `Сам объект лямбды — это экземпляр этого класса`, хранящий захваченные значения или ссылки.

### 6.3) Сколько памяти занимает пустой класс?
`Размер пустого класса (без нестатических полей) всегда не менее 1 байта`. Это гарантирует, что два разных объекта пустого класса будут иметь уникальные адреса. Однако при наследовании от пустого класса срабатывает `Empty Base Optimization (EBO)`, и базовый подобъект может не занимать дополнительной памяти в производном объекте.

### 6.4) Какие виды кастов в С++ вы знаете?
`static_cast` — явные и неявные преобразования между родственными типами (числа, иерархии классов вверх/вниз без проверки).
`dynamic_cast` — безопасное приведение вниз по иерархии с полиморфными типами с проверкой во время выполнения. 
`const_cast` — добавление/снятие `const` и `volatile` квалификаторов. 
`reinterpret_cast` — низкоуровневое переименование битового представления между несвязанными типами (указатель в число и наоборот). 
`C-style cast ((int)x)` пытается последовательно применить `const_cast`, `static_cast` с `const_cast`, `reinterpret_cast`, и потенциально опасен из-за непредсказуемости и сложности поиска в коде.

## Работа над ошибками

### 1) О лямбде (внутреннее устройство)
`Лямбда` — это синтаксический сахар, по которому компилятор генерирует анонимный класс с operator() и полями для захваченных переменных: `при захвате по значению в полях хранятся копии`, а по `ссылке — ссылки`. mutable просто убирает const с operator(), разрешая менять поля-копии, которые живут столько же, сколько сам объект лямбды. Передача лямбды в std::function по значению создаёт новую копию этого анонимного объекта со своим состоянием, поэтому изменения внутри такой копии не влияют на оригинал.

### 2) О порядке инициализации полей
`Инициализация полей класса` всегда происходит в порядке их объявления в теле класса, а не в порядке перечисления в списке инициализации конструктора. Первыми инициализируются базовые классы в порядке наследования, затем нестатические поля в порядке объявления, и только после всего этого выполняется тело конструктора. Поэтому внутри тела конструктора все поля и базовые подобъекты уже полностью сконструированы и готовы к использованию.
