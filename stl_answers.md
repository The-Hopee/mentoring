## STL

### Общие вопросы
1. **Из чего состоит STL?** — Контейнеры (хранение данных), алгоритмы (операции над диапазонами) и итераторы (связующее звено, абстракция доступа). Всё это объединено шаблонами и разбито на пространства имён.
2. **Контейнеры STL.** — Последовательные: `vector`, `deque`, `list`, `forward_list`. Ассоциативные: `map`, `set`, `multimap`, `multiset`. Неупорядоченные ассоциативные: `unordered_map`, `unordered_set` и их multi-версии. Адаптеры: `stack`, `queue`, `priority_queue`.
3. **Типы итераторов.** — Input (только чтение вперёд, `istream`), Output (только запись вперёд, `ostream`), Forward (однопроходное чтение/запись, `forward_list`), Bidirectional (двунаправленный обход, `list`, `map`), Random Access (произвольный доступ за O(1), `vector`, `deque`). Contiguous (C++17) гарантирует непрерывное расположение в памяти (`vector`, `array`).

### std::vector
1. **Устройство под капотом.** — Непрерывный динамический массив, хранящий три указателя: `begin` (начало данных), `end` (конец занятых элементов), `end_capacity` (конец выделенной памяти). При реаллокации выделяется новый блок (обычно ×1.5 или ×2), все элементы перемещаются, старый блок освобождается.
2. **Сложность операций.** — Доступ по индексу O(1), вставка/удаление в конце амортизированное O(1), вставка/удаление в середине O(n) (сдвиг элементов), поиск O(n).
3. **Добавление/удаление 101-го элемента.** — При добавлении элемента сверх capacity происходит реаллокация: выделяется новый блок (обычно 150–200 элементов), все 100 копируются/перемещаются, старый блок удаляется. При удалении 101-го элемента (`pop_back`) вызывается деструктор элемента, `end` смещается, capacity не уменьшается.
4. **std::deque vs std::vector.** — `vector` выбирают для частого доступа по индексу и вставок только в конец (непрерывная память, cache-friendly). `deque` — когда нужны быстрые вставки/удаления и в начале, и в конце (двусторонняя очередь), но не в середине, и допустимо некэш-локальное расположение блоков.

### std::forward_list
1. **Устройство.** — Односвязный список: каждый узел хранит значение и указатель на следующий узел, контейнер хранит указатель на голову. Вставка/удаление происходит изменением указателей, направление обхода — только вперёд.
2. **Сложность операций.** — Вставка/удаление после известной позиции O(1), вставка/удаление до заданной позиции O(n) (нет указателя на предыдущий узел), поиск O(n).

### std::map
1. **Устройство.** — Самобалансирующееся красно-чёрное дерево поиска (обычно), узлы содержат `std::pair<const Key, Value>`, левый потомок меньше, правый больше, высота дерева логарифмически ограничена.
2. **Сложность операций.** — Вставка, поиск, удаление — все O(log n). Итерация по отсортированному порядку ключей O(n).
3. **Итераторы после перестроения.** — Итераторы и ссылки на элементы **не инвалидируются**, поскольку узлы дерева не перемещаются в памяти — меняются только указатели на потомков.
4. **Свой тип как ключ.** — Необходимо определить `operator<` для ключа (строгий слабый порядок) или передать в шаблонный параметр компаратор — функтор с `bool operator()(const T&, const T&) const`.

### std::unordered_map
1. **Устройство.** — Хеш-таблица на основе массива указателей на «ведра» (обычно односвязные списки узлов), содержащих пару ключ-значение и хеш ключа. Индекс ведра = `hash(key) % bucket_count`.
2. **Сложность операций.** — Среднее: вставка/поиск/удаление O(1) (при хорошем хеше и низком load factor), худшее: O(n) при массовых коллизиях.
3. **Отличие от map.** — `unordered_map` использует хеш-таблицу (O(1) в среднем, порядок не определён), `map` — дерево (O(log n) гарантированно, сортированный обход). При малых N или нужде в сортировке быстрее `map`, иначе `unordered_map` быстрее на больших объёмах.
4. **Хеш-функция.** — Отображает ключ в `size_t`, должна быть детерминированной и равномерно распределять значения. Хорошая даёт минимум коллизий, плохая — «кучкует» ключи в одних ведрах, деградируя производительность до O(n).
5. **Коллизия.** — Ситуация, когда `hash(k1) % N == hash(k2) % N` для разных ключей. В STL разрешается методом цепочек (chaining): элементы одного ведра хранятся в связном списке. Альтернативы: открытая адресация (линейное/квадратичное пробирование), двойное хеширование.
6. **Рехеш.** — Перестройка хеш-таблицы при превышении `max_load_factor`: увеличивается `bucket_count`, все элементы перераспределяются по новым вёдрам (перехешируются).
7. **Итераторы после рехеша.** — Все итераторы инвалидируются, так как элементы физически перемещаются в новые ведра.
8. **Свой тип как ключ.** — Нужно предоставить хеш-функцию (`struct MyHash { size_t operator()(const T&) const; }`) и оператор сравнения `operator==` (или эквивалентный функтор).

## Умные указатели

### std::unique_ptr
1. **Зачем нужен.** — Реализует уникальное владение динамическим объектом: только один `unique_ptr` указывает на объект в любой момент. При разрушении указателя (выход из области видимости, исключение) объект автоматически удаляется, реализуя RAII.
2. **Устройство.** — Хранит сырой указатель `T* ptr` и предоставляет доступ через `operator*`/`operator->`. Копирование запрещено (удалённый конструктор копирования), перемещение разрешено (передаёт владение, обнуляя источник). Кастомный deleter по умолчанию использует `delete`, может быть заменён.
3. **Размер.** — Без кастомного deleter'а равен размеру одного сырого указателя (обычно 8 байт на 64-битной системе), благодаря Empty Base Optimization для stateless deleter'а.

### std::shared_ptr
1. **Зачем нужен.** — Реализует разделяемое владение: несколько `shared_ptr` могут совместно владеть одним объектом, объект удаляется, когда последний владеющий `shared_ptr` разрушается. Полезен для графов объектов с общими ресурсами.
2. **Устройство.** — Хранит два указателя: на управляемый объект и на Control Block (общий блок управления). Control Block содержит: счётчик сильных ссылок (shared count), счётчик слабых ссылок (weak count), deleter, allocator. При копировании/уничтожении счётчики атомарно инкрементируются/декрементируются.
3. **Размер.** — Всегда два указателя (16 байт на 64-bit), плюс сам Control Block выделяется отдельно в куче (если не использован `make_shared`).

### std::weak_ptr
1. **Зачем нужен.** — Предоставляет невладеющий «слабый» доступ к объекту, управляемому `shared_ptr`, не увеличивая счётчик сильных ссылок. Используется для разрыва циклических зависимостей (два объекта взаимно ссылаются через `shared_ptr`, что вызывает утечку) и кеширования.
2. **Устройство.** — Хранит указатель на Control Block (тот же, что у `shared_ptr`). При обращении через `lock()` атомарно проверяет счётчик сильных ссылок: если > 0, возвращает валидный `shared_ptr`, иначе — пустой.
3. **Размер.** — Совпадает с `shared_ptr` (16 байт), так как хранит те же два указателя (на объект и на Control Block).

## Linux

1. **Архитектура Linux vs Windows.** — Linux — монолитное ядро, всё работает в kernel space (драйверы, ФС, сеть), с модульной подгрузкой; Windows — гибридное микроядро. Главное различие: в Linux «всё есть файл» (устройства, сокеты, процессы видны через ФС), а в Windows — через объектную модель и WinAPI.
2. **Hard link vs symbolic link.** — Hard link — вторая запись в директории, указывающая на тот же inode (нельзя на разные ФС и директории), удаление оригинала не влияет на данные. Symbolic link — отдельный файл-ссылка, хранящий путь к цели (можно на директорию и другие ФС), при удалении цели становится «битым».
3. **chmod 755.** — Устанавливает права доступа: владелец — чтение/запись/выполнение (`7 = rwx`), группа и остальные — чтение/выполнение (`5 = r-x`). Обычно используется для директорий и исполняемых файлов.
4. **Файловые дескрипторы.** — Целочисленный идентификатор открытого файла (или сокета, пайпа). Стандартные: `0` — stdin (ввод), `1` — stdout (вывод), `2` — stderr (ошибки).
5. **Механизм swap.** — При нехватке оперативной памяти ядро вытесняет редко используемые страницы памяти из ОЗУ на swap-раздел/файл на диске (swapping). При обращении к вытесненной странице происходит page fault и подкачка обратно.

## Базы данных

1. **Типы БД.** — `Реляционные` (SQL: PostgreSQL, MySQL), `документоориентированные` (MongoDB), `ключ-значение` (Redis), `графовые` (Neo4j), `колоночные` (ClickHouse). Выбор зависит от структуры данных и паттернов доступа.
2. **Индекс.** — Структура данных (обычно B-дерево или хеш), ускоряющая поиск записей ценой замедления вставки/обновления и дополнительного места. Создаётся для полей, по которым часто выполняются `WHERE`, `JOIN`, `ORDER BY`.
3. **Транзакция.** — Группа операций над БД, которая либо выполняется полностью, либо не выполняется вовсе. Необходима для сохранения целостности данных при сбоях и конкурентном доступе.
4. **ACID.** — `Atomicity` (атомарность: всё или ничего), `Consistency` (согласованность: перевод БД из одного валидного состояния в другое), `Isolation` (изоляция: параллельные транзакции не мешают друг другу), `Durability` (долговечность: зафиксированные данные не теряются при сбое).
5. **Кеширование.** — Хранение часто запрашиваемых данных в быстрой памяти (Redis, Memcached, RAM) для снижения нагрузки на БД и уменьшения задержек. Ключевые проблемы: инвалидация кеша (устаревание данных) и стратегии вытеснения (LRU, LFU).
